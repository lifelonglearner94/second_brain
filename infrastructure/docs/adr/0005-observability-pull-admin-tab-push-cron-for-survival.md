# Observability: pull-based admin tab for debugging, push-based cron for survival

Two distinct observability concerns, served by two distinct mechanisms. **Debugging** ("why did this braindump extraction fail?") is served by the frontend admin tab — a pull-based log view over backend-provided data, consistent with frontend ADR-0001 (the frontend is a view, not compute). **Survival** ("is the Brain Replica still replicating?") is served by a host-cron health check that pushes to a notification endpoint (ntfy.sh or equivalent) when Litestream replication lag exceeds a threshold or the Brain File's volume nears capacity. The push model exists because silent replication drift is the one failure class a pull model cannot catch: the user only discovers a stale replica when they open the admin tab, by which point the trust contract from ADR-0002 has already been broken. The cron entry is zero-RAM, zero-containers — one shell script and one webhook. Self-hosted Prometheus/Grafana is explicitly rejected (300MB+ RSS, Coolify-tier overhead for a single-user system); Uptime Kuma is rejected as unnecessary — the cron line covers the survival concern at lower cost. Rejected alternative: relying on the admin tab alone (silent failure gap).
