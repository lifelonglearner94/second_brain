# SHA-pinned image tags with a separate Deploy State file for instant rollback

Container images are tagged by commit SHA (e.g. `ghcr.io/.../backend:sha-abc123`), not `:latest`. `:latest` makes rollback impossible — a bad deploy overwrites the tag and the previous image is gone from the VPS after `up -d`, forcing a fail-forward cycle (revert commit → wait for GHA → rebuild → repush) that takes minutes when seconds matter. For a system whose entire value is the Brain File, a bad backend deploy that corrupts writes with no rollback is the worst-case operational scenario. SHA-pinned tags make rollback a 30-second operation: `docker compose pull` on the previous SHA (cached on the VPS) and `up -d`. The tag is injected via a separate `deploy.env` file (`BACKEND_TAG=sha-abc123`), sourced by Compose alongside the secret `.env` — this maintains ADR-0004's separation: GHA writes `deploy.env` (non-secret Deploy State, via SSH on every deploy) but is blind to `.env` (secrets, manually placed). A compromised GitHub account can control which image the VPS runs but cannot leak runtime credentials. The Compose image reference uses a fallback (`${BACKEND_TAG:-latest}`) for local dev. Rejected alternatives: `:latest` (no rollback, fail-forward only); writing tags into the secret `.env` (violates ADR-0004's GHA-blind rule).
