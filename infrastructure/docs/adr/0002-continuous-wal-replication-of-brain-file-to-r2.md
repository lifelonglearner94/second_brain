# Continuous WAL replication of the Brain File to offsite object storage

The Brain File runs in WAL mode (required by the backend's async background jobs — ontology refactors and embedding generation run concurrently with reads, per backend ADR-0003), which legally outlaws lazy backups: a naive `cp` or `rsync` of the `.db` file while a write is in flight produces a torn snapshot missing recent WAL frames. Safe online backup requires `sqlite3 .backup`, `VACUUM INTO`, or the SQLite Online Backup API. Rather than periodic safe-snapshots (RPO of hours, silent failure if cron breaks), we adopt **Litestream** as a backend sidecar — it tails the WAL and streams changes second-by-second to **Cloudflare R2** (S3-compatible, zero egress), reducing RPO to ~1 second. R2 is chosen over a Hetzner-local target so the replica survives a datacenter-wide failure (Hetzner has historically burned). The Brain File itself lives on a named Docker volume (`sqlite_data`), not a bind mount, to avoid host/container UID-GID permission pain and keep the brain portable. Losing a day of thoughts is treated as an unacceptable epistemic breach — the trust contract of a Second Brain breaks if braindumps can vanish in a VPS fire. Rejected alternatives: periodic host-cron `sqlite3 .backup` (RPO too coarse, silent cron failure); bind mount (permission fragility); Hetzner Storage Box (same-provider failure domain).
