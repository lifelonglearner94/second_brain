# Continuous WAL replication of the Brain File to offsite object storage

The Brain File runs in WAL mode (required by the backend's async background jobs — ontology refactors and embedding generation run concurrently with reads, per backend ADR-0003), which legally outlaws lazy backups: a naive `cp` or `rsync` of the `.db` file while a write is in flight produces a torn snapshot missing recent WAL frames. Safe online backup requires `sqlite3 .backup`, `VACUUM INTO`, or the SQLite Online Backup API. Rather than periodic safe-snapshots (RPO of hours, silent failure if cron breaks), we adopt **Litestream** as a backend sidecar — it streams the WAL second-by-second to **Cloudflare R2** (S3-compatible, zero egress), reducing RPO to ~1 second. R2 is chosen over a VPS-local target so the replica survives a datacenter-wide failure (VPS has historically burned). The Brain File itself lives on a named Docker volume (`sqlite_data`), not a bind mount, to avoid host/container UID-GID permission pain and keep the brain portable. The Litestream sidecar mounts that volume **read-write** at `/data`, not read-only: Litestream v0.5.x maintains its own internal tracking tables (`_litestream_seq`, `_litestream_shadow`) in the source SQLite database to record WAL position, so a read-only mount crash-loops with `attempt to write a readonly database (8)` as soon as real R2 credentials are present. This is safe — Litestream appends only to its own `_litestream_*` tables and never mutates braindump/concept/edge rows, so the Brain File's user-data integrity is not at risk; the trust contract (RPO ~1s, off-provider replica in R2) is unchanged. Losing a day of thoughts is treated as an unacceptable epistemic breach — the trust contract of a Second Brain breaks if braindumps can vanish in a VPS fire. Rejected alternatives: periodic host-cron `sqlite3 .backup` (RPO too coarse, silent cron failure); bind mount (permission fragility); VPS Storage Box (same-provider failure domain); read-only Brain File mount (Litestream v0.5.x cannot record WAL position without writing its `_litestream_*` tracking tables, so replication never reaches steady state).
