# Deploy pipeline: build on GHA, push to GHCR, VPS pulls via command-restricted SSH key

Container images build on GitHub Actions runners and push to GHCR; the VPS never compiles. A Rust release build of an Axum + petgraph + sqlite-vec + reqwest tree transiently spikes well over 2GB of RAM — running it on the 8GB VPS would compete with the live backend, the Litestream sidecar, and the in-process Brain File for memory, risking OOM-kills of the running brain mid-compile. GHA runners are free ephemeral compute; the VPS only ever executes `docker compose pull`. Deploys are triggered by push to `main`: GHA builds, pushes images to GHCR, then SSHs into the VPS using a dedicated ed25519 deploy key whose public key is locked down in `authorized_keys` with `command="docker compose -f /opt/second-brain/docker-compose.yml pull && docker compose -f /opt/second-brain/docker-compose.yml up -d",no-pty,no-port-forwarding,no-agent-forwarding`. The command restriction *is* the security model: if the GHA private key leaks, an attacker can only restart existing containers — they get no shell, no port forwarding, no agent forwarding. Rejected alternatives: building on the VPS via Compose `build:` context (OOM risk to the running brain); an unrestricted deploy key (shell access blast radius on key leak).
