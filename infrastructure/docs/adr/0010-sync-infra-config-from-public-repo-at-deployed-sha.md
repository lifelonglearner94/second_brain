# Sync infra config from the public repo at the deployed SHA on every deploy

`bootstrap.sh` installs `docker-compose.yml`, `infrastructure/litestream.yml`, and `infrastructure/health-push.sh` once at setup; after that they never reached the VPS again, so infra-config changes bitrotted silently — the litestream read-only-mount fix shipped in the repo (commit `b7ecc69`) but the server kept the stale `:ro` compose, Litestream crash-looped on `attempt to write a readonly database (8)`, and R2 stayed empty (no offsite Brain Replica) until caught manually. `deploy.sh` now syncs those three non-secret config files from `raw.githubusercontent.com` at the deployed commit SHA (derived from `BACKEND_TAG`'s `sha-<git>` prefix) on every deploy, so the running config always matches the running images. The repo is PUBLIC, so the fetch needs no auth and introduces no new secret on the VPS or in GHA. Fetching (rather than accepting piped content) preserves ADR-0003's invariant: an attacker with only the leaked SSH deploy key can REPLAY past configs (any real commit SHA that exists in the repo) but cannot CRAFT novel ones — the config must exist in the public repo. The three files are installed deploy-owned by `bootstrap.sh`, so `deploy.sh` (which runs as the `deploy` user via the forced command) overwrites them in place with `cp` (overwriting a file you own needs write perm on the file, not the dir); `deploy.sh` itself stays root-owned and `/opt/second-brain/` stays root-owned, so the deploy key cannot create new files there or replace the `deploy.sh` gate that validates its stdin. `deploy.env` still arrives over stdin and is still ADR-0007-whitelist-validated; only config sync is new. Rejected alternatives: pipe a config tar over stdin (an SSH-key-only attacker could craft arbitrary compose, bind-mount `.env`, and exfiltrate secrets — breaks ADR-0003's command-restriction model); a git checkout on the VPS (new moving part + a read token on the server = a new secret, rejected by ADR-0004's no-secret-on-VPS-for-deploy stance); a signed tar verified by deploy.sh (works for a private repo but adds a GHA signing-key secret + crypto to deploy.sh — unnecessary since the repo is public and fetch-from-repo achieves the same replay-only guarantee with zero new secrets); syncing `deploy.sh` itself (mid-execution chicken-and-egg — it is the root-owned gate, updated manually like `.env`); syncing `health-push.cron` (a `/etc/cron.d` system path, rarely changed, manual `bootstrap` re-run). Rejected: doing nothing (the bitrot that emptied R2 is the proof this was needed).
