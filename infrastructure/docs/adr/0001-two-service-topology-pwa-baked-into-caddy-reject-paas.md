# Two-service Compose topology with PWA baked into Caddy, rejecting PaaS

Personal-Scale: a custom Caddy image (TLS termination, `file_server` for the built PWA, `/api` reverse-proxy) and the Rust backend - two long-running services, no frontend container. The PWA Bundle is baked into the Caddy image at build time (not a shared volume) to avoid volume permission/ownership pain and keep the Edge disposable as an *image* (rebuilt per deploy). The Edge is NOT stateless at runtime: Caddy's ACME state - issued Let's Encrypt certificates and ACME account keys in `/data`, plus autosaved config in `/config` - is runtime state that must survive container recreations, so both paths are mounted on named volumes (`caddy_data`, `caddy_config`). Without `/data`, every `docker compose down/up` or deploy wipes ACME state and forces a fresh cert request, which trips Let's Encrypt's 5-issuances-per-168h rate limit and takes HTTPS down (#55). ACME account keys live in this volume - same host-disk trust boundary as the Brain File (`sqlite_data`), NOT the Zero-Trust Image boundary (ADR-0004): the volume is host-side runtime state, nothing is baked into the image. PaaS options (Coolify/Dokploy) are explicitly rejected: their control plane (Traefik + PostgreSQL + Redis + workers) steals RAM from sqlite-vec and the backend on an 8GB VPS, introducing fleet-management overhead for a personal-scale system with a handful of invite-gated users. Raw `docker-compose.yml` + a ~5-line Caddyfile + a GitHub Actions SSH-deploy script give absolute control with zero idle overhead.
