# Zero-Trust Image contract: .env.example source-of-truth + CI guard

ADR-0004 established the secret model (`.env` on the host, GHA-blind, Zero-Trust Images) as a guideline. This ADR makes it an enforceable contract with two artifacts. First, `infrastructure/.env.example` is the single source of truth for the full runtime-secret key list — values blank, grouped by purpose (Gemini, R2/Litestream, Health Push/ntfy, WebAuthn/config), each key marked `[SECRET]` or `[config]`. Later slices consume keys from the host `.env`; they do not extend the contract ad hoc. Second, a CI guard (`infrastructure/test/zero-trust.sh`, run in the `zero-trust` GHA job) asserts that no `ENV`, `ARG`, or `COPY` directive in any Dockerfile in the repo bakes a secret literal — a secret-named key with a non-placeholder value, a long dense base64/hex literal under any name, or a `COPY` of the real `.env` / a key file — so images remain public-safe artifacts: a leaked GHCR image exposes only compiled code. The guard is self-testing (`infrastructure/test/zero-trust-test.sh`): it plants a secret in a throwaway Dockerfile and asserts the guard fails, then asserts the guard passes on the real backend + Edge Dockerfiles. The real `infrastructure/.env` remains manually placed on the VPS over SSH once and is never read, written, or referenced by CI — GitHub Actions is blind to runtime secrets (it ships only code and Deploy State per ADR-0007). No secret manager is introduced (Doppler/Infisical/Vault remain rejected by ADR-0004 as Coolify-tier overhead).
