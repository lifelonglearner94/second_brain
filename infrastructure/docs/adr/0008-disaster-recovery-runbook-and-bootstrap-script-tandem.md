# Disaster recovery: runbook as human source of truth, bootstrap script as automation, tested periodically

Catastrophic VPS loss is the one scenario where the brain must come back from R2 under stress, at a bad time, executed exactly once — the worst possible conditions for improvisation. Recovery is served by two artifacts working in tandem. **`infrastructure/DISASTER_RECOVERY.md`** is the human-readable runbook — the source of truth for the why and the exact command sequence. **`infrastructure/bootstrap.sh`** automates the deterministic drudge work (installing Docker, configuring the firewall, placing the cron health-check script + ntfy webhook). The runbook is intentionally short: (1) run `bootstrap.sh` on a fresh VPS, (2) manually place `.env` (ADR-0004 — GHA is blind to secrets), (3) run `litestream restore` from R2 into the named volume, (4) `docker compose pull && up -d`. The separation mirrors the deploy model: the script handles everything reproducible, the human handles only the two steps that require secrets or judgment. An untested restore procedure is a hope, not a strategy — the procedure must be exercised on a throwaway VPS periodically, because ADR-0002's trust contract is only as strong as the last confirmed restore. Rejected alternatives: runbook only (human error under stress); bootstrap script only (no human-readable why); improvisation (Personal-Scale is not an excuse for undocumented manual processes).
