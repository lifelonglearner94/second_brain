# One named write-intent exception to "strictly a view"; the Service Worker stays dumb

ADR-0001 establishes the frontend as strictly a view over the backend. The frontend holds persistent state in three forms, and the distinction between them is load-bearing: (1) the cached Global Topology Snapshot in IndexedDB is a *read cache* — a frozen copy of backend truth for offline resilience, consistent with being a view; (2) Viewport State in LocalStorage is *frontend-owned UI state* the backend has no concept of; (3) Pending Captures in IndexedDB is *write-intent* — state the backend does not know about, representing the user's committed intent to submit a braindump that has not yet crossed the network boundary. Form (3) is the sole, explicitly named exception to "the frontend holds no persistent state of its own." By naming and bounding the single exception, the rule is fortified rather than eroded: any future proposal to add persistent frontend state must either qualify as a read cache or as UI state, or it must supplant this ADR — it cannot accrete silently.

The Service Worker caches the app shell (HTML/JS/CSS/assets) only. It does not intercept API calls, manage the offline write queue, or hold business logic. The frontend application code explicitly manages saving the Global Topology Snapshot to IndexedDB, the Pending Captures queue, and Viewport State in LocalStorage. The Service Worker is notoriously difficult to debug; keeping it a pure static-asset cache keeps all offline logic in the application layer where it is testable and inspectable.

## Considered options

- **Justify Pending Captures as "not really state"** (argue it's a transient network buffer, not a true exception): rejected — mental gymnastics that erode the principle by pretending the exception doesn't exist. Naming it explicitly is more honest and more durable.
- **Service Worker intercepts and queues API calls** (the "standard" PWA offline pattern — SW intercepts POST /ingest when offline and replays on reconnect): rejected — moves business logic (the review-and-confirm requirement from ADR-0007's explicit-submit rule) into the SW layer, which is hard to debug, hard to test, and runs in a separate execution context from the UI. The Pending Captures review flow requires UI interaction (the user must correct offline STT and confirm); a SW-initiated replay would bypass that. App-layer ownership of the queue keeps the review-and-confirm logic where the user can see it.

## Consequences

- The frontend's persistent-state inventory is exactly three items, each with a named home: Global Topology Snapshot (IndexedDB, read cache), Viewport State (LocalStorage, UI state), Pending Captures (IndexedDB, write-intent — the sole exception). Any fourth item requires either reclassification into one of these three categories or a new ADR.
- Offline behavior: the app loads the Frozen Graph (cached snapshot + staleness indicator), chat is disabled, and new captures route to Pending Captures for review-and-confirm on reconnect. The user is never shown a blank screen for a connectivity failure — the app always renders *something*.
- The Service Worker is a static-asset cache and nothing more. All offline logic (snapshot persistence, capture queuing, viewport restoration) lives in the application layer. This makes the offline story debuggable in the browser's main thread and testable with standard frontend tooling.
