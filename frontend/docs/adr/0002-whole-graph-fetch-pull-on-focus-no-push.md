# Whole-graph fetch on load, pull-on-focus for deltas, no real-time push

> **Reopened (#43):** the canonical Global Topology Snapshot is **store-held**, not page-local. A single deep `GraphStore` (Svelte 5 runes store, `src/lib/state/graph.svelte.ts`) owns the snapshot, the delta-sync cursor, the derived `GraphData`, and every mutation against them (`loadFromNetworkOrCache`, `syncDelta`, `mergeIngest`, `mergeEndorsedEdge`, `applyConceptMerge`, `applyTypeMerge`). The app page, the housekeeping page, and the endorsements page all read from this one store - none re-fetch the Global Topology Snapshot. The previous arrangement smeared the snapshot across a page-local `snapshot` + cursor in the app page, a re-fetch inside `HousekeepingStore` for label resolution, and a re-fetch on the endorsements page; the shallow `spatial-graph.svelte.ts` singleton held only the derived `GraphData`. That singleton is absorbed into `GraphStore`. The decision below - one whole graph in memory for the session, pull-on-focus deltas, no push - stands unchanged; only the wording tightens to match the store-held implementation.

The frontend fetches the entire knowledge graph (the Global Topology Snapshot) in one compressed JSON call on bootstrap and holds it in the `GraphStore` (and thus the Spatial View-Graph) for the session. Cache invalidation is pull-on-focus: a Delta Sync (changes since last_sync_timestamp) is fetched whenever the PWA regains window focus, and braindump ingestion responses are optimistically merged into the store. There is no WebSocket, no SSE, no polling loop - the backend remains a stateless Axum server. This is licensed by two project-wide constraints: Personal-Scale (a prolific user of ~10 concepts/day takes a decade to reach 35k concepts, well under the 50k-node ceiling at which WebGL + ForceAtlas2 hold 60fps on mobile - whole-graph fetch and in-memory layout are non-issues at V1 scale), and Per-User Isolation (issue #72: each user's graph is scoped by `user_id` and only mutated by that user themselves, so within a user's graph there is no collaborating actor to push deltas from; cross-user graphs are isolated and never visible to each other, so brief visual staleness between focus events harms nothing).

## Considered options

- **Windowed / k-hop subgraph fetch** (page in a neighbourhood around the user's focus, expand on navigation): rejected at V1 - introduces windowing logic, partial-graph ForceAtlas2 (which behaves badly on subgraphs because it lacks the global topology that produces organic clusters), and a navigation-driven fetch latency that feels worse than a one-time bootstrap download. Reconsider only if the user crosses the 50k-concept ceiling, which is a non-issue for a V1 personal-scale app.
- **Real-time push** (WebSocket / SSE streaming deltas from backend to frontend): rejected - forces the backend to maintain persistent connections and track per-client state, abandoning the stateless-Axum property, solely to save the user from a few minutes of visual staleness on background mutations (e.g. async ontology retags - backend ADR-0003). Per-user isolation (issue #72) makes the collaborating-actor problem that push solves a non-problem: within a user's graph the only actor is that user, and cross-user graphs are never shared.
- **Background polling** (periodic re-fetch on a timer): rejected - burns bandwidth and battery at rest for no benefit over pull-on-focus, and still isn't real-time. Pull-on-focus gets the same freshness guarantee at a fraction of the cost because the user is the only actor and they are the one re-focusing the window.

## Consequences

- The `GraphStore` holds the whole graph in memory for the session. This is a deliberate non-optimization: at Personal-Scale it is cheaper to hold 35k nodes than to implement windowing, and it makes ForceAtlas2's global-layout output coherent (a subgraph layout is not a preview of the global layout - it is a different picture). Because the snapshot is store-held, every page that needs it (app, housekeeping, endorsements) reads the same in-memory copy - there is no per-page re-fetch of the Global Topology Snapshot, and no second copy of the graph for label resolution.
- A braindump submission does not wait for background sync - the POST /ingest response returns the newly extracted nodes and edges, and `GraphStore.mergeIngest` optimistically merges them into the snapshot and rebuilt `GraphData` for immediate visual confirmation. The next Delta Sync reconciles any divergence (e.g. if the backend's extraction differed from the optimistic merge).
- **Staleness window**: an async ontology refactor (backend ADR-0003) that retags an edge while the user is staring at it will not be reflected until the next Delta Sync (next window focus or next ingestion). This is accepted: a stale `[affects]` edge shown for five minutes while the backend has already upgraded it to `[endangers]` does no semantic damage - the user is not acting on edge types in real-time, and the next focus event overwrites it.
- **Flip condition**: if the user ever crosses ~50k concepts, the whole-graph fetch and in-memory layout assumptions break, and this ADR is superseded by a windowing strategy. The 50k bound is the load-bearing number; until then, windowing is premature complexity.
